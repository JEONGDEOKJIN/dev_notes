

## 이 그림을 설명할 것 임 
![Image](https://i.imgur.com/JjDq3Mm.png)

```bash
- 우선, form 제출에 대해서, '다양한 상태' 가 존재한다고 느꼈음 
- '상태를 분석' 할 수 있는 '툴' 로써, '상태 다이어그램' 이 존재한다는 것을 알게 되었음 -> 그래서, 1) 이걸 기반으로 form 에 일반적으로 요구되는 상태를 파악하고 2) '리액트 제어 컴포넌트' 와 'React Hook Form' 의 비제어 방식을 이해하고자 함 -> 이걸 통해 React Hook Form 에 좀 더 쉽게 이해하고 사용하고자 함. 

```

### 1. 정지 상태 

```
[상태 설명]
- 사용자가 쓸 수 있는 상태 여야 함 

[리액트 제어 컴포넌트]
    - input 의 name 에 '해당 input 의 고유한 이름' 을 할당함 
        ex) <input name = "customer_input">
    - '신뢰 가능한 단일 출처(single source of truth)' 설정 
        - input 태그의 value 로 설정 
        - 이걸 무엇으로 보고, 어떤 메커니즘으로 다루냐에 따라서 방식이 달라지는 것 같음 (#⭐⭐⭐)

[React Hook Form]
    - register 메서드에 인자를 전달 
    - destructuring 을 진행하는데, 구체적인 로직을 아직 모르겠음 (#📛📛📛)


```

### 2. 작성중 
```
[상태의 의미]
    - 사용자가 input 에 값을 입력 
    - 실시간으로 해당 값이 신뢰의 원천에 업데이트 될 수 있어야 함

[리액트 제어 컴포넌트]
    1) 변화가 되면 -> onChange 를 발동 시킴 -> onChange 에 사전에 바인딩 시킨 useState 의 set 함수가 onChange 가 건네주는 data 를 받아옴 -> 저장하고, useState 의 state 로 전달함 
    2) state 는 input 의 value 에 사전에 꽂혀 있음 
    3) 그래서, onChange 가 발동하면 -> useState setState 를 거쳐서 -> value 에 바인딩 되는 식으로 update 됨 

[React Hook Form]
    1) destructuring 에 의해 내부 로직에 의해 처리

```

### 3. 제출 
```
[상태의 의미]
    - 사용자가 작성을 완료하고 최종 제출을 함 
    - 이후, 제출된 데이터를 '검증' 하게 됨.

[리액트 제어 컴포넌트]
    - 버튼에 onSubmit 이벤트를 달고, 해당 이벤트 핸들러를 발동 
    - 그러면, form 의 속성에 붙어 있는 이벤트 핸들러가 실행 
    - onSubmit 이벤트 함수가 실행됨 
        1) 해당 함수에서 유효성 검사 
        2) 맞으면 -> 성공 로직 
        3) 틀리면 -> 에러 로직을 실행 (이때, 에러 문구 표기)

[React Hook Form]
    - onSubmit 의 첫 번째 인자에 성공 함수, 두 번째 인자에 에러 함수를 넘긴다. 

```

### 4. '유효성 검사' 성공/실패
```
[상태의 의미]
    - '제출 이벤트' 가 발생해서 '유효성 검사' 실행 후 -> 통과 가 된 상태 

[리액트 제어컴포넌트]
    - 바로 통신을 보낸다. 
    - 이후 '통신 상태 정상 여부' 를 분기 처리 해야 한다. 

[React Hook Form 비제어 컴포넌트]
    - 마찬가지로, 통신을 하고, 통신 정상 여부를 처리한다. 
    - 이게 내부 로직으로 동작
```
